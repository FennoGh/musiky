## 2. Local setup

Requires Node 20+, Python 3.10+, PostgreSQL 15.

```bash
# 1. Create the local DB
createuser -s musiky
createdb musiky -O musiky
psql -d postgres -c "ALTER USER musiky PASSWORD 'musiky_dev'"

# 2. Backend deps
cd backend && python3 -m venv .venv && .venv/bin/pip install -r requirements.txt && cd ..

# 3. Frontend deps
cd frontend && npm install && cd ..

# 4. Root deps (just `concurrently`)
npm install

# 5. Env
cp .env.example .env        # then edit DATABASE_URL / DB_PASSWORD / secrets

# 6. Schema + demo data
npm run db:migrate
npm run db:seed

# 7. Run web + api together
npm run dev
```

| URL | What |
|---|---|
| http://localhost:3000 | Web app |
| http://localhost:8000/health | API health probe |

For mobile dev: `./bin/dev-mobile.sh` (auto-detects laptop LAN IP, rewrites `frontend/.env.local` and `capacitor.config.ts`, restarts both servers, rebuilds + installs the Android APK on a connected device).

## 3. Backend (Django + DRF)

### 3.1 Authentication

- **Scheme:** JWT via `djangorestframework-simplejwt`. HS256, 7-day access token (configurable via `ACCESS_TOKEN_EXPIRE_MINUTES`).
- **Login:** `POST /auth/login` accepts `{email, password}` *or* OAuth2 form `{username, password}` for back-compatibility. Returns `{access_token: str, token_type: "bearer"}`.
- **Register:** `POST /auth/register` accepts `{email, password, name?}`. Returns the same token shape, `201`.
- **Custom user model:** `apps.users.models.User` (declared via `AUTH_USER_MODEL = "users.User"`). Email is the `USERNAME_FIELD`. Primary key is a CUID string (32 chars), generated by `apps.users.utils.ids.cuid()` to match the legacy schema.
- **Frontend storage:** `localStorage.musiky_token`. On 401, the token is cleared and the user is redirected to `/login`.

### 3.2 Permissions

Implicit, project-membership-based (no RBAC table). Two helpers in `apps/projects/permissions.py`:

| Helper | Behavior |
|---|---|
| `project_for_member(project_id, user)` | Returns `(project, is_owner)` if the user owns OR collaborates on the project; raises `404` otherwise (existence is never leaked to non-members). |
| `project_for_owner(project_id, user)` | Returns the project if the user owns it; raises `403` otherwise. |

DRF default authentication is `JWTAuthentication`; default permission is `IsAuthenticated`. Public endpoints (`/health`, `/auth/*`, GET `/platforms`) opt out via `@permission_classes([AllowAny])`.

### 3.4 Critical business logic

#### Residual-split allocation (`Collaborator` add / update / remove)

Splits must always sum to `100.00`. Rather than asking the user to manually rebalance, the system maintains the **owner's row as the residual**:

| Action | Effect on owner's row |
|---|---|
| Add collaborator with split S | If `total + S > 100`, take `S - (100 - total)` from the owner. If owner doesn't have enough → `400 Total split would exceed 100%`. |
| Increase a non-owner's split by Δ | Take Δ from the owner. If owner has < Δ → `400 Owner only has X% to give`. |
| Decrease a non-owner's split by Δ | Return Δ to the owner. |
| Remove a non-owner | Return their full split to the owner. |
| Edit owner's role / split directly | Refused — owner role is locked, owner split is computed. |

All these mutations run inside a single `transaction.atomic()` block with `select_for_update()` on the owner row to prevent races between concurrent adds.

#### Revenue → Payout fan-out

When the owner posts `POST /projects/{id}/revenues`:

1. Validate platform exists, amount > 0.
2. Insert `Revenue`.
3. For each `Collaborator`:
   - Compute `share = round(amount * split_pct / 100, 2)` using `ROUND_HALF_UP`.
   - Create one `Payout` row.
   - **Owner's payout is created `PAID`** (the platform's deposit already arrived to the owner; nothing to "transfer to themselves").
   - **Other collaborators' payouts are `PENDING`**.
4. **Rounding remainder** (`amount - sum_of_shares`) is added to the owner's payout so the total reconciles exactly.
5. Insert an `Activity` row of type `REVENUE_RECEIVED`.

All steps are inside one transaction.

When editing/deleting an existing Revenue:

- If any non-owner Payout linked to it has `status=PAID`, the request is **refused with 409** (don't silently erase money that already moved).
- Otherwise, delete linked Payouts, mutate the Revenue, and re-fan-out from the new amount (only when amount changed).

This is the part the project actually exists to get right — see `apps/finance/views.py:_build_payouts` and `:_external_paid`.

### 3.5 File uploads

- Saved to `MEDIA_ROOT/{covers,audio}/{cuid}.{ext}` via 1 MB chunked write with running size check.
- MIME-type whitelist: `image/jpeg|png|webp|gif` for covers, `audio/mpeg|mp3|wav|x-wav|ogg|mp4|x-m4a|flac` for audio.
- Size limits: 8 MB images, 60 MB audio.
- In dev: served via Django's `static()` helper. In prod: `nginx` serves `/uploads/` directly from the bind-mounted volume.

### 3.6 CORS

Regex allowlist in `settings.CORS_ALLOWED_ORIGIN_REGEXES`:

- `localhost`, `127.0.0.1`
- RFC1918 ranges: `192.168.x.x`, `10.x.x.x`, `172.16–31.x.x` (LAN dev with the Android device on the same Wi-Fi)
- The hardcoded production IP `174.138.1.11`

`CORS_ALLOW_CREDENTIALS = True`. All methods and headers are allowed.

### 4.2 API client

`frontend/lib/api.ts` exports:

- `apiFetch<T>(path, init?)` — wraps `fetch` with `Authorization: Bearer <token>`, JSON content-type defaulting, automatic `clearToken()` on 401, error-message extraction from the `{detail}` response shape.
- `apiAsset(path)` — resolves an upload URL (`/uploads/covers/abc.jpg`) into an absolute URL using the API base.
- `getToken()` / `clearToken()` — `localStorage.musiky_token`.

The base URL is `process.env.NEXT_PUBLIC_API_BASE ?? 'http://127.0.0.1:8000'` — baked into the JS bundle at build time.

### 4.3 Auth guard pattern

Each protected page does:
```tsx
useEffect(() => {
  if (!getToken()) router.replace('/login')
  apiFetch<UserMe>('/me').catch(() => {/* 401 already cleared */})
}, [router])
```
There is no global middleware; every page is its own gate. Sign-out (`DashboardHeader`) calls `clearToken()` then `router.replace('/login')`.

## 5. Data model

Nine domain tables. Money columns are `Decimal(12, 2)`; split percentages are `Decimal(5, 2)`. Every table has a CUID string PK (32 chars) for URL/route compatibility with the legacy schema.

| Table | Key fields | Notable constraints |
|---|---|---|
| `User` | id, email (unique), name, image, plan, password (PBKDF2) | `email` is the login field |
| `Project` | id, title, cover_url, status, owner→User, created_at, released_at | Status in `{DRAFT, READY, LIVE, ARCHIVED}`. `LIVE` blocks edits to non-status fields |
| `Track` | id, project→Project, title, file_url, cover_url, version, duration, uploaded_at | Cascade delete on Project |
| `Collaborator` | id, project→Project, user→User, role, split_pct, joined_at | `UNIQUE(project, user)`. Owner row's split is the residual |
| `Activity` | id, project→Project (nullable), actor→User (nullable), type, payload (JSON), created_at | Index `(project, -created_at)` |
| `Expense` | id, project→Project, payer→User, category, amount, currency, description, spent_at | |
| `Platform` | id, name (unique), slug (unique) | Auto-slugged on create |
| `Distribution` | id, project→Project, platform→Platform, status, live_at, streams | `UNIQUE(project, platform)` |
| `Revenue` | id, project→Project, platform→Platform, amount, currency, period_start, period_end, received_at | Index `(project, -received_at)` |
| `Payout` | id, project→Project, revenue→Revenue, user→User, amount, status, paid_at | Index `(user, status)`, `(project)` |

For the visual ERD, see [ERD.md](./ERD.md) and [ERD.png](./ERD.png).

## 6. Mobile (Capacitor)

The Android app is a thin Capacitor WebView pointed at the live Next.js URL. There is no native code, no separate mobile API surface, and no static export — the `webDir: "frontend/public"` setting exists only as a Capacitor formality; the app loads the URL specified in `server.url`.

`bin/dev-mobile.sh`:
1. Detects the laptop's current LAN IP via `ip route get 1.1.1.1`.
2. Rewrites `frontend/.env.local` (`NEXT_PUBLIC_API_BASE=http://<ip>:8000`).
3. Rewrites `capacitor.config.ts` (`server.url=http://<ip>:3000`).
4. Updates `android/app/src/main/res/xml/network_security_config.xml` to whitelist cleartext for that IP.
5. Restarts Django on `0.0.0.0:8000` and Next on `0.0.0.0:3000`.
6. If the IP is unchanged since last run **and** an APK exists, skips the rebuild and just relaunches.
7. Otherwise: `npx cap sync` → `gradlew assembleDebug` → `adb install -r` → `adb shell monkey -p dev.musiky.app …`.

Pass `FORCE=1` to override the skip-rebuild shortcut.

## 7. Docker / production

### docker-compose

Three services on two isolated bridge networks:

```
networks:
  backend:   db ↔ api
  frontend:  api ↔ web

services:
  db    postgres:15-alpine    no ports exposed
  api   build ./backend       :8000  → backend + frontend networks, gunicorn 2 workers
  web   build ./frontend      :3000  → frontend network only, next start (standalone)
```

Healthchecks on all three (`pg_isready`, `curl /health`, `wget /`). Memory limits: 200 MB db, 220 MB api, 180 MB web.

### Build details

- **api Dockerfile** (`backend/Dockerfile`): `python:3.12-slim` base, installs build-essential temporarily for psycopg, runs migrations on container start (`migrate --noinput`), then `gunicorn musiky.wsgi:application --bind 0.0.0.0:8000 --workers 2 --timeout 90`.
- **web Dockerfile** (`frontend/Dockerfile`): two-stage. Stage 1 (`node:20-alpine`) installs deps with `npm ci` and runs `next build`. Stage 2 copies the standalone bundle, exposes `:3000`, runs `node server.js`. No `node_modules` ship in the runner image.

### Production deploy

The live VPS uses Docker Compose (rebuilds from the same `Dockerfile`s). `infra/deploy.sh` rsyncs the working tree to `/opt/musiky/`, rebuilds images, restarts containers with `docker compose up -d --build`, then polls `/health` for 60 seconds and rolls back on failure.

Other ops scripts:

| Script | What |
|---|---|
| `infra/init_server.sh` | Debian 12 droplet bootstrap — apt updates, swap, Docker CE, UFW, git clone |
| `infra/backup.py` | `pg_dump` + tar of uploads volume |
| `infra/logs_collect.py` | Periodic journalctl snapshots |
| `infra/port_audit.sh` | External + internal port visibility |
| `infra/cost_monitor.sh` | DigitalOcean billing snapshot |
| `infra/locustfile.py` | Load test (login → list → detail) |

## 9. Where things are not yet wired up

Honest list — things the codebase shapes but doesn't fully implement:

- **Billing** is stubbed. `/billing/*` endpoints return mock subscription/invoice data; no Stripe integration.
- **Notifications** preferences (`PUT /me/notifications`) accept the request but don't persist.
- **Distributions** are immediately marked `LIVE` on creation. Real distributors take hours to confirm; we treat the API call as the source of truth.
- **Stream counts** are entered manually via `PATCH /distributions/{id}/streams`. No automated platform polling.
- **Background jobs / queue / webhooks**: none. Every request is synchronous.

These are intentional: the project is the operational ledger, not a full distribution provider.
